## 一、專案目標（大方向）

最初的問題是「如何串 LINE 訊息自動回覆」，但經過釐清，真正要做的是一套**個人內容飛輪**：

> 選題雷達 → 與 AI 協作撰寫 → 發佈到個人部落格 → 擴散到社群/LINE → 回流實體影響力（課程、顧問、講座）

LINE 自動回覆只是這個飛輪最尾端的一個觸點，不是核心。核心是「選題 → 撰寫 → 部落格」這前三段。

### 四層架構

- **選題雷達（半自動）**：排程抓 RSS / Hacker News / arXiv，用 Claude API 過濾評分，產出「今日值得寫的主題清單」推給本人勾選。不自動往下寫。
- **協作撰寫（本人主導）**：AI 做苦工（蒐集、初稿、查證），本人做判斷（觀點、取捨、定調）。這層刻意不全自動——個人影響力來自「本人的觀點」，不是 AI 量產。
- **部落格（已開始建）**：Astro + Netlify，純靜態、近零成本，文章即 Markdown，對齊既有 GitHub 工作流。
- **擴散與回流**：新文章自動推 LINE / 社群，文末放課程/顧問入口。LINE 自動回覆（含 RAG 問答）放在這一層才合理。

## 二、關鍵決策（含理由）

### 定位

- 「給專業工作者看的科技新知」，純科技策展為主體、不限本業。
- 隱含價值：幫讀者過濾雜訊 + 講清楚「這跟你的工作有什麼關係」。
- **重要提醒（待消化）**：純科技策展是紅海，中文圈已有科技新報、INSIDE、TechNews 等強者。建議每篇至少帶一點「本人視角」（職安 × 大學教學 × AI 實務的交叉經驗），這是 AI 內容農場無法複製的護城河。技術上零成本，只差 AI 撰稿時的提示詞設定。本人保有最終決定權。

### 產能

- 目標一週 2-3 篇（積極經營）。
- 待觀察的張力：純策展 + 高頻產出，三個月後易遇「寫得勤但沒人看」的撞牆期。帶入個人視角可緩解。

### 視覺設計：方向 A

- 暖紙底 + 墨綠編輯感（像一本好雜誌，Stratechery 路線）。
- 否決的另兩個方向：B 近黑底+青藍（科技銳利但偏冷、長文較累），C 微灰白+單橘（極簡但記憶點弱）。
- 選 A 理由：在「想讀下去」上最強，暖底耐讀、墨綠有編輯感卻不冷淡，容錯率最高。

### 字型

- 標題：Noto Serif HK Black（900 字重） — 氣勢。
- 內文：Noto Sans TC Regular（400） — 長文好讀。
- 襯線標題 + 無襯線內文，是現代內容站的黃金組合。

### 封面圖

- 程式自動生成的色塊封面（非找圖、非 AI 算圖）。
- 每篇依分類自動生成，零檔案、零版權、風格統一。
- 各分類配色：AI=深墨綠、工具=深褐金、產業=深藍灰、技術解析=深莓。

## 七、重要原則（避免重蹈覆轍）

- **備份才是命根**：這次原碼、備份 zip、Node 一起消失，靠這份文件才救回來。程式碼一定要推上 GitHub（已完成），重要文件用純文字（`.md`）納入版控。
- **解壓縮陷阱**：zip 重複解壓 / 解到不同位置會造成「多包一層」或結構打散。解壓後務必確認「專案根目錄 = 有 `package.json` 的那層」。
- **AI 內容定位**：AI 做苦工、本人做最終把關。不追求全自動量產，個人觀點才是護城河。
- **平台風險**：部落格（自有網域）才是真正擁有的資產；LINE / 社群是租來的流量。架構以部落格為核心。
- **字型載入**：Noto Serif HK Black 檔案大，首次載入標題可能閃一下才變粗，屬正常，之後快取。已設中文系統字型 fallback；CJK 字型分片多，不用 preload。
